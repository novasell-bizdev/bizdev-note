# note記事作成

朝会の発表+議論からnote記事ドラフトを一気通貫で生成するオーケストレータースキル。

チーム定義: `.claude/agents/note-article-team.md`

## 引数
$ARGUMENTS — 朝会テーマ or 資料ファイルパス。省略時は `materials/morning-sessions/` の最新資料を使用

オプション:
- `draft-only` — Phase 3（執筆）まで。販促素材は生成しない
- `promo-only` — 既存記事に対して Phase 5（販促素材）のみ実行

## 実行手順

### Phase 0: 素材整理

1. `$ARGUMENTS` から入力素材を特定:
   - ファイルパス指定時: そのファイルを読み込み
   - テーマ指定時: `materials/morning-sessions/` から該当資料を検索
   - 省略時: `materials/morning-sessions/` の最新ファイルを使用
2. 発表内容と議論メモを分離（素材内に議論メモが含まれる場合）
3. `materials/team-strategy/` からチーム方針資料も参照（整合性確認用）

### Phase 1: リサーチ（並列3エージェント）

**リサーチャーA（note同テーマ調査）** — Agent(Explore):
- note.com で同テーマの記事を調査
- どのような切り口・構成が人気かを分析
- 差別化ポイントを特定

**リサーチャーB（最新情報収集）** — Agent(Explore):
- テーマに関する最新情報をWeb検索で収集
- 統計データ・引用可能なソースを収集（GEO最適化: AI引用率40%+向上）
- 業界動向・事例を収集

**リサーチャーC（チーム方針整合性確認）** — Agent(Explore):
- `materials/team-strategy/` の資料を読み込み
- 記事内容がチーム方針と矛盾しないか確認
- 方針資料から引用・言及できるポイントを抽出

### Phase 1.5: 連載コンテキスト読み込み

**連載の記憶を呼び起こすステップ。** 過去記事との継続性・差別化を確保する。

1. `published/content-registry.json` を読み込み:
   - 既存記事のタイトル・テーマ・キー論点を把握
   - 前回記事の `keyQuestions`（未解決の問い）を抽出
   - 前回記事の `keyTopics` と今回テーマの重複・発展関係を整理

2. 前回記事の本文を読み込み（`content-registry.json` の `file` パス）:
   - トーン・文体の一貫性を確認
   - 「次回予告」があればその約束を確認
   - 使用した関係者引用を把握（同じ引用の重複を避ける）

3. 連載コンテキストレポートを生成:
   ```
   ## 連載コンテキスト
   - 前回記事: #N「タイトル」
   - 前回の未解決の問い: [リスト]
   - 前回の次回予告: [あれば]
   - 今回拾うべき論点: [前回の問いのうち今回触れるもの]
   - 避けるべき重複: [前回と被るトピック・引用]
   - 連載番号: #N+1
   ```

### Phase 2: 構成案作成 → ユーザー承認ゲート

リサーチ結果 + 連載コンテキストを統合し、`templates/article-template.md` に沿って構成案を作成:

- タイトル案（30-35文字、シリーズ名は本文冒頭に記載。タイトル自体にはシリーズ名を含めない）
- H2/H3見出し構成
- 各セクションの要点（3行以内）
- **前回の問いへの応答**（該当する場合）: どの問いを拾い、どう展開するか
- カテゴリ・タグ候補
- 差別化ポイント
- 連載上の位置づけ（前回→今回→次回の流れ）

→ **ユーザーに構成案を提示して承認を得る**

### Phase 3: 原稿執筆

構成案に基づいて執筆。

**トーン**:
- 一人称: 「私たち」（チームの公開日誌）
- 文体: 「です・ます」ベース
- 文字数: 1,500～2,000字

**パンチライン設計（冒頭・締め・本文）**:
- **冒頭（3段落以内）**: 以下の3型から1つ選択
  - エピグラフ型: ダッシュ付き1行の宣言 → リアルな現場の声
  - 自己論駁型: 過去の常識を引き → 現実で否定
  - 挑発的問い型: 業界常識に正面からケンカを売る断言
- **締め（最後の2-3段落）**: 以下の2型から1つ選択
  - 意志宣言型: 「私たちは～だ」の一人称宣言
  - 覚悟+問い型: 自分への戒め → 読者も巻き込む問い
- **本文中**: 各セクションに1つ以上のパンチライン
  - 逆説・再定義（「XではなくYだ」）
  - 短文断言（格言的、140文字以内でXに引用可能）
  - 具体→抽象の跳躍（泥臭い数字 → 哲学的命題）
- **品質基準**: 冒頭/締めのパンチラインを `[PUNCHLINE]` でマーキング

**連載としての接続**:
- 冒頭で前回記事への言及（該当する場合）: 「前回は～について書きました」
- 前回の未解決の問いを拾う場合: 該当セクションで自然に言及
- 末尾の「次回予告」で次のテーマを示唆

**構成ルール**:
- 冒頭120字: メタディスクリプション代替。キーワード＋要約＋ベネフィット
- H2/H3にキーワード配置
- リスト・テーブルを活用（GEO最適化）
- 統計データ・数値を含める（AI引用率40%+向上）
- 「結論が出なかった問い」セクション必須
- 800-1000文字ごとに画像挿入ポイント `[IMAGE: 説明]`
- 過去記事への内部リンク `[LINK: 記事タイトル]`

**SEO執筆ルール**:
- 冒頭200文字以内に主要KW
- h2にサブKW
- 1段落3-5行（スマホ可読性）
- 1文60文字以内目安
- 太字・箇条書き・引用で視覚リズム

**段落の緩急**: 長い説明の直後に一行の短文断言を挟む。口語と文語のミックスで固さを崩す

**禁止事項**:
- 戦略方針・顧客情報・財務数値の記載
- 社内事例の具体社名・具体数値（抽象化する）
- 「朝会」への言及（シリーズ名は「BizDev 公開日誌」）

`draft-only` 指定時はここで完了。Phase 6 に進む。

### Phase 3.5: 関連リンク提案（Cross-Linker）

Phase 3 完了後、cross-linker エージェントを自動呼び出し:
- ドラフトファイルのパスを引数に渡す
- link-registry.md からの内部リンク + WebSearch による外部リンクを提案
- ユーザーが採用した提案をドラフトに反映してから Phase 4 へ進む

### Phase 4: 並列サブエージェント呼び出し

以下を並列実行:

1. **article-review** サブエージェント — 品質チェック（5軸20項目 + 機密チェック + 連載整合性）
2. `/title-optimizer` スキル — タイトル6パターン生成
3. `/thumbnail-gen` スキル — サムネイル3パターン提案

### Phase 5: 統合・修正

1. article-review の評価が合格基準未達の場合: 修正指示に基づき原稿を修正（最大2回）
2. cross-linker のリンク提案を本文に埋め込み
3. title-optimizer の結果をユーザーに提示し、タイトルを確定
4. thumbnail-gen の結果をユーザーに提示

`promo-only` 指定時はここから開始（既存記事パスを指定）。

### Phase 5.5: 販促素材生成

`/sns-teaser` スキルを呼び出し:
- X告知文3パターン + Xスレッド + LinkedIn告知文
- 投稿スケジュール
- note概要文

### Phase 6: 保存 + 連載データ更新

1. 記事原稿を `articles/drafts/{NN}_{slug}.md` に保存

2. `published/content-registry.json` に新規エントリーを追加:

```json
{
  "id": "N",
  "slug": "slug-name",
  "title": "タイトル",
  "seriesNumber": "N",
  "date": "YYYY-MM-DD",
  "author": "発表者名",
  "category": "カテゴリ",
  "tags": ["#AIBizDev"],
  "noteUrl": "",
  "file": "articles/drafts/{NN}_{slug}.md",
  "summary": "記事の要旨（2-3文）",
  "keyQuestions": ["この記事で提示した未解決の問い"],
  "keyTopics": ["主要トピック3-5個"],
  "linkedFrom": [],
  "linkedTo": []
}
```

3. cross-linker の逆方向リンク提案があれば、過去記事の末尾リンクセクションを更新

4. **次のステップガイドをユーザーに提示**:

```
## ドラフト完了！次のステップ

1. 品質チェック結果を確認 → 修正があれば対応
2. `git add` + `git commit` → PR作成
   - ブランチ名: `article/{NN}-{slug}`
   - PRテンプレートのチェックリストを埋める
3. 馬場（編集長）がレビュー → Approve待ち
4. noteに公開:
   - 記事本文をコピー＆ペースト
   - サムネイル画像（1280x670px）を設定
   - タグ5つを設定: {タグリスト}
   - マガジン「ノバセル｜bizdev note」に追加
   - 推奨投稿時間: 水-木 12:00 or 20:00-21:00
5. 公開後タスク:
   - noteのURLを content-registry.json に記入
   - 前の記事（#N-1）の末尾に今回記事へのリンクを追記
   - `/sns-teaser` でSNS告知文を生成 → X/LinkedIn/Slackで拡散
```

## エラー対応

| エラー | 対応 |
|--------|------|
| 会議録なし・テーマ不明 | ユーザーにテーマ or 素材の提供を依頼 |
| KW競合過多 | ロングテールKW or 独自切り口での差別化を提案 |
| 機密情報検出 | 即停止、該当箇所を明示してユーザーに確認 |
| 文字数過少（1,500未満） | 素材不足を報告、追加素材 or テーマ拡張を提案 |
| Quality Gate 3回FAIL | ユーザーにエスカレーション、手動修正を依頼 |
