# クロスリンカーサブエージェント

note-article スキルから呼び出され、自チーム過去記事＋関係者記事との関連リンクを提案するサブエージェント。
Hub&Spoke + Chain型のリンク設計パターンを適用する。

## 入力

- 新規記事のドラフト（Markdown）
- 新規記事のメタ情報（カテゴリ・タグ・スラッグ）

## 実行手順

### Step 1: レジストリ読み込み

1. `published/content-registry.json` を読み込み — 自チーム過去記事のメタデータ・要旨・キー論点
2. `research/link-registry.md` を読み込み — 関係者noteアカウント・記事リスト

### Step 2: マッチング

以下の基準で関連度をスコアリング:

**自チーム過去記事（content-registry.json）**:
| 基準 | 重み |
|------|------|
| keyTopicsの重複 | 最高 |
| keyQuestionsとの接続 | 高 |
| 同一カテゴリ | 中 |
| タグの共通数 | 中 |
| 連載の前後関係 | 高（直前・直後は自動リンク） |

**関係者記事（link-registry.md）**:
| 基準 | 重み |
|------|------|
| リンク推奨度（A/B/C） | 最高 |
| 本文キーワードの重複 | 中 |
| 時系列の近さ | 低 |

### Step 3: リンク提案の生成

以下の4種類の提案を生成:

#### A. 連載チェーンリンク（自動）

直前・直後の記事への相互リンクは自動提案:
- 冒頭: 「前回の公開日誌では～について書きました」
- 末尾: 「次に読む」に前回記事

#### B. 本文中挿入提案

記事本文のどの位置に、どの記事（自チーム or 関係者）へのリンクを配置すべきか。

```
## 本文中リンク提案

1. [挿入位置: H2「XXX」の後] → [リンク先記事タイトル]
   導入文: 「この問題の詳細は、以下の記事で具体例とともに解説しています。」
   形式: テキストリンク / カードリンク
```

#### C. 記事末尾「関連記事」

関連度の高い順に最大3本を提案。

#### D. 逆方向リンク提案

過去記事のどこに新記事へのリンクを追記すべきか。

```
## 逆方向リンク提案（過去記事への追記）

1. [過去記事タイトル] の [挿入位置]
   追記文: 「このテーマの続編として、[新記事タイトル] でさらに議論しています。」
```

### Step 4: 初期フェーズ対応（記事0-2本時）

レジストリの記事が0-2本の場合、リンク提案の代わりに以下を出力:

```
## 今後リンクすべきテーマ提案

現在の記事本数が少ないため、相互リンクの提案は縮退運転します。
以下のテーマで記事を書くと、今回の記事と相互リンクできます:

1. [提案テーマ] — 関連理由: ...
2. [提案テーマ] — 関連理由: ...
```

## リンク設計パターン

### Hub&Spoke + Chain型

```
[ハブ記事（シリーズまとめ）]
     │
     ├─ #01 → #02 → #03 ...  （Chain: 次→前の順送り）
     │
     ├─ #04 → #05 ...           （別シリーズ）
     │
     └─ 各記事からハブへの双方向リンク
```

### リンク数の目安

| 記事本数 | 1記事あたりのリンク数 |
|----------|---------------------|
| 0-5本 | 2-3本（ハブ + 前後） |
| 5-10本 | 4-6本（ハブ + 前後 + 関連1-2） |
| 10本+ | 5-8本（ハブ + 前後 + 関連2-3 + サブハブ） |

## 出力形式

```
## クロスリンクレポート

### レジストリ状況
自チーム記事: N本
関係者記事: M本
フェーズ: Phase 1 / Phase 2 / Phase 3

### A. 連載チェーンリンク
...

### B. 本文中リンク提案
...

### C. 記事末尾 関連記事
...

### D. 逆方向リンク提案
...

### E. content-registry.json 更新案
[新規記事のエントリー + linkedFrom/linkedTo の更新]
```
