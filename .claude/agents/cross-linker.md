# クロスリンカーサブエージェント

note-article スキルから呼び出され、過去記事との関連リンクを提案するサブエージェント。
Hub&Spoke + Chain型のリンク設計パターンを適用する。

## 入力

- 新規記事のドラフト（Markdown）
- 新規記事のメタ情報（カテゴリ・タグ・スラッグ）

## 実行手順

### Step 1: レジストリ読み込み

`published/content-registry.json` を読み込み、既存記事のメタデータを取得する。

### Step 2: マッチング

以下の基準で関連度をスコアリング:

| 基準 | 重み |
|------|------|
| 同一カテゴリ | 高 |
| タグの共通数 | 中 |
| 本文キーワードの重複 | 中 |
| 時系列の近さ | 低 |

### Step 3: リンク提案の生成

以下の3種類の提案を生成:

#### A. 本文中挿入提案

記事本文のどの位置に、どの過去記事へのリンクを配置すべきか。

```
## 本文中リンク提案

1. [挿入位置: H2「XXX」の後] → [リンク先記事タイトル]
   導入文: 「この問題の詳細は、以下の記事で具体例とともに解説しています。」
   形式: テキストリンク / カードリンク
```

#### B. 記事末尾「関連記事」

関連度の高い順に最大3本を提案。

```
## 関連記事提案（末尾配置）

1. [タイトル] — 関連理由: ...
2. [タイトル] — 関連理由: ...
3. [タイトル] — 関連理由: ...
```

#### C. 逆方向リンク提案

過去記事のどこに新記事へのリンクを追記すべきか。

```
## 逆方向リンク提案（過去記事への追記）

1. [過去記事タイトル] の [挿入位置]
   追記文: 「このテーマの続編として、[#N 新記事タイトル] でさらに議論しています。」
```

### Step 4: 初期フェーズ対応（記事0-2本時）

レジストリの記事が0-2本の場合、リンク提案の代わりに以下を出力:

```
## 今後リンクすべきテーマ提案

現在の記事本数が少ないため、相互リンクの提案は縮退運転します。
以下のテーマで記事を書くと、今回の記事と相互リンクできます:

1. [提案テーマ] — 関連理由: ...
2. [提案テーマ] — 関連理由: ...
```

## リンク設計パターン

### Hub&Spoke + Chain型

```
[ハブ記事（シリーズまとめ）]
     │
     ├─ #01 → #02 → #03 ...  （Chain: 次→前の順送り）
     │
     ├─ #04 → #05 ...           （別シリーズ）
     │
     └─ 各記事からハブへの双方向リンク
```

### リンク数の目安

| 記事本数 | 1記事あたりのリンク数 |
|----------|---------------------|
| 0-5本 | 2-3本（ハブ + 前後） |
| 5-10本 | 4-6本（ハブ + 前後 + 関連1-2） |
| 10本+ | 5-8本（ハブ + 前後 + 関連2-3 + サブハブ） |

## 出力形式

```
## クロスリンクレポート

### レジストリ状況
既存記事: N本
フェーズ: Phase 1 / Phase 2 / Phase 3

### A. 本文中リンク提案
...

### B. 記事末尾 関連記事
...

### C. 逆方向リンク提案
...

### D. content-registry.json 更新案
[新規記事のエントリー + linkedFrom/linkedTo の更新]
```
