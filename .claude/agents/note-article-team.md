# note-article チーム定義

BizDev公開日誌の議論をnote記事として制作する6エージェント構成のチーム。
SEO/LPO/SNS拡散を統合的に設計し、「検索で見つかり、読まれ、シェアされる」記事を生成する。

---

## エージェント一覧

| Agent | 役割 | 担当Phase |
|-------|------|----------|
| Material Curator | 素材整理・論点抽出 | Phase 1 |
| SEO Strategist | KW選定・タグ戦略・見出し構造 | Phase 1 |
| Structure Designer | 構成設計・LPO・CTA配置 | Phase 2 |
| Article Writer | 記事執筆 | Phase 3 |
| Quality Gate | SEO/LPO/機密/ブランド/SNS拡散力の多軸品質チェック | Phase 4 |
| Promotion Generator | X告知・スレッド・OGP・投稿スケジュール | Phase 5 |

---

## Agent詳細

### Material Curator

**役割**: 会議素材を記事執筆に使える形に整理する

**入力**:
- 会議録テキスト（Google Meet議事録 or 手動貼り付け）
- 公開日誌スライド（PPTXの内容 or 要約）
- 補足資料（イベントメモ、参考記事など）
- 過去の記事（シリーズの文脈理解用）

**処理**:
1. 会議録から「AI×事業開発」に関連する論点を抽出（LP制作等の個別案件話は除外）
2. 各論点を以下の構造で整理:
   - **主張**: 何が言われたか（1文）
   - **根拠**: なぜそう言えるか（データ/経験/事例）
   - **対立意見**: 反対意見や別の視点があったか
   - **示唆**: BizDev/読者にとっての意味
3. 引用可能なデータ・数字を特定（独自データは被リンク獲得の核）
4. 過去記事との関連性・差分を明確化

**出力**: `material_pool.md`
- 論点リスト（優先度付き、5-8個）
- 引用可能データリスト
- 過去記事との接続ポイント
- 記事テーマ候補（2-3案）

---

### SEO Strategist

**役割**: note.comのSEO特性を活かしたキーワード・タグ・見出し戦略を設計する

**入力**:
- 記事テーマ（Material Curatorの出力 or ユーザー指定）

**処理**:
1. **KW調査**（WebSearch）:
   - 主要キーワードの検索ボリューム推定
   - note.com内の競合記事調査（同テーマの上位記事の構成・文字数・スキ数）
   - 関連キーワード・共起語の収集
2. **タグ戦略**:
   - 安定タグ × 上昇タグの組み合わせ（5-7個）
   - noteのお題企画との適合性チェック
3. **見出し構造の提案**:
   - h2にサブKWを含める設計
   - 見出しだけで記事全体の流れが把握できるか検証
4. **タイトル候補**:
   - 26-40文字、KW前半配置、数字入り、ベネフィット明示
   - 3案生成

**出力**: `seo_strategy.md`
- 主要KW + サブKW + 共起語リスト
- タグ候補（5-7個、安定/上昇の分類付き）
- タイトル候補3案（SEOスコア付き）
- 推奨見出し構造（h2/h3の階層）
- 競合分析サマリー（上位3記事の構成・差別化ポイント）

---

### Structure Designer

**役割**: SEO戦略とLPO設計を統合した記事構成を設計する

**入力**:
- `material_pool.md`（Material Curator）
- `seo_strategy.md`（SEO Strategist）
- ユーザーのHITL Gate 1フィードバック

**処理**:
1. **記事構成設計**:
   - 論点の取捨選択・順序決定
   - 各セクションの文字数配分（全体3,000-5,000文字目安）
   - 見出しの最終決定（SEO Strategistの推奨を反映）
2. **LPO設計**:
   - スクロール深度マップ:
     ```
     [FV] アイキャッチ + 冒頭3行フック
       ↓
     [目次] 全体の見通し
       ↓
     [Section 1] 最重要論点（離脱ポイントを超えさせる）
       ↓
     [中間CTA] フォロー誘導 or マガジン誘導
       ↓
     [Section 2-3] 本論深掘り
       ↓
     [中盤まとめ] ここまでのポイント
       ↓
     [Section 4+] 応用・発展・未解決の問い
       ↓
     [まとめ] 結論再提示 + 読者への問いかけ
       ↓
     [末尾CTA] フォロー + シェア + マガジン + 次回予告
     ```
   - CTA配置計画（中間CTA / 末尾CTA の具体文言）
   - 有料/無料判定（有料の場合は30-40%地点にライン設計）
3. **読者との対話設計**:
   - オープンクエスチョンの設計（記事末尾）
   - コメント誘導の仕掛け

**出力**: `article_structure.md`
- 完全な見出し構造 + 各セクション要約（3行以内）
- CTA配置マップ
- 有料/無料判定 + 理由
- スクロール深度設計図
- 読者への問いかけ案

### Article Writer

**役割**: 構成に基づいて記事を執筆する。冒頭と締めにパンチラインを配置し、SNS拡散力を最大化する。

**入力**:
- `article_structure.md`（Structure Designer）
- `material_pool.md`（素材参照）
- `seo_strategy.md`（KW密度の目安）
- ユーザーのHITL Gate 1フィードバック

**トーンガイド**:
- **主語**: 「私たち」「チーム」（個人の意見ではなくチームの議論として）
- **文体**: 「です・ます」ベース。やや硬めだが、公開日誌のカジュアルさも残す
- **具体性**: 「チームの議論で出た話」「ある案件での経験」など実務ベース
- **禁止**: クライアント名、具体的な予算額、社内の人事情報、未公開プロダクト情報
- **E-E-A-T**: 実務経験に基づく記述を各セクションに1つ以上含める

**パンチライン設計（SNS拡散の核）**:

冒頭と締めに「そのままXに引用されるフレーズ」を埋め込む。田部構文を参考にした5つの修辞技法を使い分ける。

***冒頭（最初の3段落以内）— 3つの型から選択***:

| 型 | 構造 | 例 |
|----|------|-----|
| **エピグラフ型** | ダッシュ付き1行の宣言 → 直後にリアルな現場の声を引用 | `――再現性なき情熱に、未来は宿る。` → 「最近の人は…」 |
| **自己論駁型** | 過去の自分/常識を引き → 現実で冷徹に否定 | 「1年前、私たちはこう考えていた。…で、結果はどうだったか？」 |
| **挑発的問い型** | 業界の常識に正面からケンカを売る断言 or 問い | 「多くの企業で、AIは『ゴミを量産する装置』にしかなっていない。」 |

***締め（最後の2-3段落）— 2つの型から選択***:

| 型 | 構造 | 例 |
|----|------|-----|
| **意志宣言型** | 「私たちは～だ」の一人称宣言で締める | 「私たちはまだ、登り始めたばかりだ。」 |
| **覚悟+問い型** | 自分への戒め → 読者も巻き込む問い | 「それくらいの基準を持てる組織だけが、次の切符を手にする。」 |

***本文中のパンチライン技法（各セクションに1つ以上）***:

| 技法 | パターン | 例 |
|------|---------|-----|
| **逆説・再定義** | 「XではなくYだ」で既知の概念をひっくり返す | 「作業を仕事と呼んでいた。これからは意志を仕事と呼ぶ。」 |
| **短文断言** | 1-2文の格言的断定。主語+述語の最小構造 | 「アウトプットに価値はない。成果だけが存在証明になる。」 |
| **メタファー一貫** | 1つの比喩を記事全体で展開し切る | マシンガン→照準→シューター→引き金 |
| **具体→抽象の跳躍** | 泥臭い数字・エピソード → 一気に哲学的命題へ | 「500件営業した」→「何かを起こすなら常に狂気的な熱狂がそこにあった」 |
| **格言の一点投入** | 先人の言葉を1箇所だけ、ここぞの場面で使う | 「勝ちに不思議の勝ちあり、負けに不思議の負けなし。」 |

***パンチラインの品質基準***:
- 140文字以内で完結する（Xにそのままコピペできる長さ）
- 対比構造で「あるある」を否定する（リツイート衝動）
- 主語が「私たち/我々」で読者を巻き込む
- 具体→抽象の振れ幅がシェア欲を刺激する

**SEO執筆ルール**:
- 冒頭200文字以内に主要KWを含める
- h2にサブKWを自然に含める
- 1段落3-5行（スマホ可読性）
- 1文60文字以内を目安
- 太字・箇条書き・引用で視覚的リズム
- 画像挿入ポイントを `[IMAGE: 説明]` で指示
- 内部リンク候補を `[LINK: 過去記事タイトル]` で指示

**段落内の緩急設計**:
- 長い説明段落の直後に「だが、あえて問いたい。」等の一行段落を挟む
- 口語と文語のミックスで固さを崩す（「猫も杓子もAIだ」等の口語を時折混ぜる）
- 数字の具体性で信頼を獲得し、直後に抽象的な信念を語る

**出力**: `article_draft.md`
- メタデータ（タイトル、ステータス、文字数、タグ）
- 本文（SEOルール + パンチライン設計 適用済み）
- CTA（中間 + 末尾、構成設計通り）
- `[PUNCHLINE]` タグ: 冒頭・締め・本文中の主要パンチラインをマーキング（Quality Gateでチェック用）

### Quality Gate

**役割**: 多軸での品質チェック。全軸PASSで次Phase、FAILがあればArticle Writerに差し戻し

**入力**: `article_draft.md`

**チェック軸**:

#### 軸1: SEOスコア（5項目）
| ID | チェック | 基準 |
|----|---------|------|
| S1 | タイトル | 26-40文字、KW前半、数字 or ベネフィットあり |
| S2 | 冒頭KW | 200文字以内に主要KW出現 |
| S3 | 見出し構造 | h2にサブKW、h2→h3の階層崩れなし |
| S4 | 本文品質 | 2,000-5,000文字、段落3-5行、太字/箇条書き使用 |
| S5 | タグ | 5-7個、安定×上昇の組み合わせ |

#### 軸2: LPOスコア（4項目）
| ID | チェック | 基準 |
|----|---------|------|
| L1 | FV設計 | 冒頭3行で価値提示 + ターゲット明示 |
| L2 | CTA配置 | 中間CTA 1箇所以上 + 末尾CTA（フォロー/シェア/マガジン） |
| L3 | スクロール設計 | 15-20行ごとに小見出し、中盤まとめあり |
| L4 | 対話設計 | 末尾にオープンクエスチョン |

#### 軸3: 機密・コンプライアンス（3項目）
| ID | チェック | 基準 |
|----|---------|------|
| C1 | クライアント情報 | 社名・予算・契約内容の漏洩なし |
| C2 | 社内機密 | 未公開プロダクト・人事情報・戦略方針・財務数値・投資判断の漏洩なし |
| C3 | 引用・著作権 | 引用元の明記、画像の権利確認 |

#### 軸4: ブランド・トーン（4項目）
| ID | チェック | 基準 |
|----|---------|------|
| B1 | チームの声 | 「私たち」視点が一貫、個人の独断に見えない |
| B2 | 議論プロセス | 結論だけでなく「対立→気づき→統合」のプロセスが見える |
| B3 | E-E-A-T | 各セクションに実務経験ベースの記述が1つ以上 |
| B4 | 語調統一 | 記事全体でです/ます調 or だ/である調が一貫。引用・発言部分は例外 |

#### 軸5: SNS拡散力（4項目）
| ID | チェック | 基準 |
|----|---------|------|
| X1 | 冒頭パンチライン | 最初の3段落以内に「常識を否定する断言」or「自分ごと化する問い」がある |
| X2 | 締めパンチライン | 最後の2-3段落が「意志/覚悟の一人称宣言」で終わる。分析・提言で終わっていない |
| X3 | 引用可能フレーズ | 本文中に140文字以内で完結し、対比構造を持つパンチラインが2つ以上ある |
| X4 | 概念再定義 | 既知の概念に新しい意味を与えている箇所が1つ以上ある |

**判定ルール**:
- 軸1-2: 各軸で1つでもFAIL → 差し戻し（修正指示付き）
- 軸3: 1つでもFAIL → **即停止**、ユーザーに報告
- 軸4: 2つ以上FAIL → 差し戻し
- 軸5: 2つ以上FAIL → 差し戻し（パンチライン強化指示付き）
- 全軸PASS → 「Quality Gate: PASS」と報告
- **最大2回の差し戻し**。3回目のFAILはユーザーにエスカレーション

**出力**: `quality_report.md`
- 各軸のPASS/FAIL + 具体的な指摘
- 差し戻し時の修正指示

### Promotion Generator

**役割**: 記事公開後の拡散を最大化する販促素材を生成する

**入力**:
- `article_draft.md`（Quality Gate PASS済み）
- `seo_strategy.md`（タグ・KW参照）

**処理**:
1. **X告知文**（3パターン）:
   - **問い型**: 読者の共感を呼ぶ問いかけ → 記事URLで回答
   - **気づき型**: 意外な数字/事実 → 解釈 → 記事URL
   - **リスト型**: 要点3-5個のチラ見せ → 記事URL
   - 各140文字以内、ハッシュタグ2-3個
2. **Xスレッド設計**（5-7ツイート）:
   - 記事の要点をスレッドで展開
   - 最終ツイートに記事URL
   - スレッド単体でも価値がある内容
3. **OGP画像指示書**:
   - 画像サイズ: 1200x630px
   - 含めるべきテキスト（タイトル短縮版）
   - 色・フォント・レイアウトの指示
4. **投稿スケジュール**:
   - 記事公開タイミング（平日7-8時 or 20-22時推奨）
   - X告知: 公開同時
   - X再投稿: 翌日（別切り口）
   - リマインド: 1週間後
5. **note概要文**: 140字以内（検索結果表示用）

**出力**: `promotion_kit.md`
- X告知文3パターン
- Xスレッド（5-7ツイート）
- OGP画像指示書
- 投稿スケジュール
- note概要文

## 品質基準サマリー

| 観点 | 目標 |
|------|------|
| SEO | タイトル26-40文字、KW密度適切、見出し階層正、タグ5-7個 |
| LPO | FV3行価値提示、中間+末尾CTA、15-20行ごとに見出し |
| 機密 | クライアント名・予算・契約内容の漏洩ゼロ |
| ブランド | 「私たち」視点、議論プロセス可視化、E-E-A-T充足、語調統一 |
| SNS拡散力 | 冒頭パンチライン、締め意志宣言、引用可能フレーズ2+、概念再定義1+ |
| 文字数 | 3,000-5,000文字（質優先、水増し禁止） |
| 拡散 | X告知3パターン + スレッド + 投稿スケジュール |
